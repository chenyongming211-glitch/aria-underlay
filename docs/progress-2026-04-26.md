# Aria Underlay 开发进度汇报 2026-04-26

本文记录 2026-04-26 截止当前的整体开发进度、事务性合规状态、今日完成事项、已知风险和下一步计划。

## 1. 当前总体结论

`aria-underlay` 当前方向没有偏离原计划。项目已经从“方案设计”进入“可验证的事务骨架 + Python Adapter fail-closed 边界 + CI 覆盖”的阶段。

当前主线仍然是：

```text
Rust Underlay Core
  -> gRPC / Protobuf
  -> Python Underlay Adapter
  -> NETCONF / Mock Backend
  -> Switch Management Endpoint
```

架构边界保持清晰：

- Rust 负责产品模型、设备纳管、intent、planner、diff、事务状态机、journal、recovery、InDoubt、telemetry event。
- Python Adapter 负责设备连接、NETCONF/NAPALM/CLI 后端、厂商 renderer、设备侧 prepare/commit/verify/recover。
- 单 management endpoint 是当前 ACID 事务边界。
- 多交换机场景按多个 endpoint 独立事务处理，不再宣传跨设备强原子性。

## 2. 当前阶段定位

当前处于 Sprint 1 后段到 Sprint 2 前段的交界：

- Sprint 0 capability probe 已形成可运行路径。
- Sprint 1A mock adapter、事务状态、失败路径、journal phase 覆盖已经比较完整。
- Sprint 1B real NETCONF backend 的准备工作已经开始，真实设备联调前的 fail-closed 骨架已落地。
- Sprint 2 renderer skeleton 已启动，但 Huawei/H3C renderer 仍明确标记为 skeleton，不能进入生产下发路径。

### 3.2 真实 NETCONF prepare 的 fail-closed 边界

已完成：

- `NcclientNetconfBackend.prepare_candidate()` 已接入真实设备侧 prepare 骨架：
  - `lock(candidate)`
  - render candidate XML
  - `edit-config(candidate)`
  - `validate(candidate)`
  - 失败后 `discard-changes`
  - 最后 `unlock(candidate)`
- renderer 缺失时返回结构化错误，不会进入假成功。
- skeleton renderer 被真实 backend 拒绝，不允许下发未验证 XML。

价值：

- 没有真实交换机时，也能先把“不能假成功、不能乱下发”的边界锁住。
- 真实设备到位后，只需要补 vendor XML 和设备差异，不需要重写事务边界。

### 3.3 Renderer registry fail-closed

已完成：
