# Aria Underlay 开发进度汇报 2026-04-26

本文记录 2026-04-26 截止当前的整体开发进度、事务性合规状态、今日完成事项、已知风险和下一步计划。

## 1. 当前总体结论

`aria-underlay` 当前方向没有偏离原计划。项目已经从“方案设计”进入“可验证的事务骨架 + Python Adapter fail-closed 边界 + CI 覆盖”的阶段。

当前主线仍然是：

```text
Rust Underlay Core
  -> gRPC / Protobuf
  -> Python Underlay Adapter
  -> NETCONF / Mock Backend
  -> Switch Management Endpoint
```

架构边界保持清晰：

- Rust 负责产品模型、设备纳管、intent、planner、diff、事务状态机、journal、recovery、InDoubt、telemetry event。
- Python Adapter 负责设备连接、NETCONF/NAPALM/CLI 后端、厂商 renderer、设备侧 prepare/commit/verify/recover。
- 单 management endpoint 是当前 ACID 事务边界。
- 多交换机场景按多个 endpoint 独立事务处理，不再宣传跨设备强原子性。

## 2. 当前阶段定位

当前处于 Sprint 1 后段到 Sprint 2 前段的交界：

- Sprint 0 capability probe 已形成可运行路径。

### 3.12 真实 NETCONF scoped state filter 骨架

已完成：

- `NcclientNetconfBackend.get_current_state(scope)` 已接入 scoped running 读取路径。
- 空 scope 返回空状态并跳过设备读取，避免没有变更范围时产生无意义 RPC。
- full scope 执行 `get-config(source="running")`，但真实 parser 未完成前仍返回 `NETCONF_STATE_PARSE_NOT_IMPLEMENTED`。
- 非空 scoped scope 会构造 subtree filter，并执行 `get-config(source="running", filter=("subtree", xml))`。
- `verify_running(desired_state, scope)` 复用同一 scoped running 读取路径；读取后仍返回 `NETCONF_VERIFY_NOT_IMPLEMENTED`，不会假装验证成功。
- `StateScope.vlan_ids` 会排序、去重并校验 VLAN 范围。
- `StateScope.interface_names` 会排序、去重、XML escape，并拒绝空接口名。

关键测试：

- full scope 不带 filter 读取 running。
- 空 scope 不触碰设备。
- scoped VLAN/interface filter 会排序、去重、转义。
- 非法 VLAN scope 返回 `INVALID_STATE_SCOPE`。
- parser/verify 未实现时仍显式 fail-closed，不返回假状态或假成功。

价值：

- 真实设备到位前，已经固定 Rust -> adapter -> NETCONF `get-config` 的 scope 边界。
- 后续只需要补厂商 running state parser 和 vendor-specific filter/profile，不需要重做 gRPC 或事务调用链。
- touched subtree refresh / post-commit scoped verify 的性能优化有了协议层落点，同时不牺牲正确性。

### 5.1 2026-04-30 事务可靠性修复进度

本轮先处理不依赖真实交换机的 ACID/事务可靠性问题，修复方案见：

- [Transaction Reliability Hardening Plan](./superpowers/plans/2026-04-30-transaction-reliability-hardening.md)

已落地：

- `Committed` journal 现在只在 shadow state 成功写入后写入，关闭“journal terminal 但 shadow 未落盘”的主要崩溃窗口。
- shadow 写入失败会把事务标记为 `InDoubt`，不再返回 `SuccessWithWarning`。
- 多设备聚合状态不再把“部分成功 + 部分失败/回滚”报告成成功警告。
- recovery 批量路径在恢复前重新读取 journal 记录，避免使用 list 阶段的过期记录继续恢复。
- journal 增加 `error_history`，保留阶段性错误历史，不再只剩最后一个错误。
- NETCONF prepare 出错后，如果 discard 也失败，会保留原始 prepare 错误并附加 discard 失败细节。
- mock NETCONF 端口模式 normalize 对未知模式 fail-closed，不再静默当成 access。
- ncclient 错误分类去掉裸 `"auth"` 子串匹配，避免把 authorization 文本误判为 authentication failure。

验证状态：

- Python adapter 全量测试已在本地通过：`188 passed`。
- `git diff --check` 已通过。
- 当前 shell 缺少 `cargo`/`rustc`，Rust 本地测试无法执行。
- GitHub Actions `25165739879` 已通过 Rust `cargo check`、Rust `cargo test`、Python adapter tests 和集成矩阵。
