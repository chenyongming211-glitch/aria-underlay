# Aria Underlay 开发进度汇报 2026-04-26

本文记录 2026-04-26 截止当前的整体开发进度、事务性合规状态、今日完成事项、已知风险和下一步计划。

## 1. 当前总体结论

`aria-underlay` 当前方向没有偏离原计划。项目已经从“方案设计”进入“可验证的事务骨架 + Python Adapter fail-closed 边界 + CI 覆盖”的阶段。

当前主线仍然是：

```text
Rust Underlay Core
  -> gRPC / Protobuf
  -> Python Underlay Adapter
  -> NETCONF / Mock Backend
  -> Switch Management Endpoint
```

架构边界保持清晰：

- Rust 负责产品模型、设备纳管、intent、planner、diff、事务状态机、journal、recovery、InDoubt、telemetry event。
- Python Adapter 负责设备连接、NETCONF/NAPALM/CLI 后端、厂商 renderer、设备侧 prepare/commit/verify/recover。
- 单 management endpoint 是当前 ACID 事务边界。
- 多交换机场景按多个 endpoint 独立事务处理，不再宣传跨设备强原子性。

## 2. 当前阶段定位

当前处于 Sprint 1 后段到 Sprint 2 前段的交界：

- Sprint 0 capability probe 已形成可运行路径。

### 3.9 Journal / artifact GC 最小可用版本

已完成：

- `JournalGc::run_once()` 不再是空实现。
- 按 `RetentionPolicy` 清理旧的 terminal journal：
  - `Committed`
  - `RolledBack`
  - `Failed`
- `InDoubt` 和非终态事务不会被自动 GC。
- rollback artifact 只在能从 journal 确认关联事务是 terminal 时清理。
- 支持 `max_artifacts_per_device`，按设备保留最新 terminal artifact。
- 新增 GC 单元测试覆盖：
  - 旧 terminal journal 删除。
  - InDoubt journal 保留。
  - terminal artifact 删除。
  - unknown tx artifact 保留。
  - 单设备 artifact 数量上限。

价值：

- Durability 不再只有“写入”，也有明确留存和清理策略。
- 避免长期私有化部署后 journal/artifact 无限制增长。
- 不牺牲 recovery 安全性：无法证明 terminal 的事务和 artifact 不会被 GC。

### 3.11 真实 NETCONF scoped state filter 骨架

已完成：

- `NcclientNetconfBackend.get_current_state(scope)` 已接入 scoped running 读取路径。
- 空 scope 返回空状态并跳过设备读取，避免没有变更范围时产生无意义 RPC。
- full scope 执行 `get-config(source="running")`，但真实 parser 未完成前仍返回 `NETCONF_STATE_PARSE_NOT_IMPLEMENTED`。
- 非空 scoped scope 会构造 subtree filter，并执行 `get-config(source="running", filter=("subtree", xml))`。
- `verify_running(desired_state, scope)` 复用同一 scoped running 读取路径；读取后仍返回 `NETCONF_VERIFY_NOT_IMPLEMENTED`，不会假装验证成功。
- `StateScope.vlan_ids` 会排序、去重并校验 VLAN 范围。
- `StateScope.interface_names` 会排序、去重、XML escape，并拒绝空接口名。

关键测试：

- full scope 不带 filter 读取 running。
- 空 scope 不触碰设备。
- scoped VLAN/interface filter 会排序、去重、转义。
- 非法 VLAN scope 返回 `INVALID_STATE_SCOPE`。
- parser/verify 未实现时仍显式 fail-closed，不返回假状态或假成功。

价值：

- 真实设备到位前，已经固定 Rust -> adapter -> NETCONF `get-config` 的 scope 边界。
- 后续只需要补厂商 running state parser 和 vendor-specific filter/profile，不需要重做 gRPC 或事务调用链。
- touched subtree refresh / post-commit scoped verify 的性能优化有了协议层落点，同时不牺牲正确性。

### 3.12 真实 NETCONF running state parser 注入边界

已完成：

- `NcclientNetconfBackend` 新增 `state_parser` 注入点。
- 未配置 parser 时，`get_current_state` 和 `verify_running` 仍 fail-closed，不返回假状态。
- 非 production-ready parser 会返回 `NETCONF_STATE_PARSER_NOT_PRODUCTION_READY`。
- production-ready parser 可以把 `get-config` 返回的 XML 转换为标准 observed state。
- `verify_running` 在 parser 可用时会用解析后的 observed state 对比 desired state，覆盖 VLAN、interface、description、admin state、access/trunk mode。
- parser 返回结构会做基础 shape 校验，避免非 dict / 缺少字段导致静默假成功。

价值：

- 真实设备 parser 还没写之前，生产路径继续关闭。
- 下一步实现 Huawei/H3C parser 时，只需要实现 `parse_running(xml, scope)`，事务层、driver 层和 gRPC 契约不需要重做。
- post-commit verify 已经具备“读取 running -> parser -> desired 对比”的完整骨架。

### 3.13 State parser registry 与厂商 skeleton

已完成：

- 新增 `state_parsers` 模块。
- Huawei/H3C running state parser skeleton 已登记。
- parser registry 默认拒绝 skeleton parser，返回 `STATE_PARSER_NOT_PRODUCTION_READY`。
- Cisco/Ruijie/Unknown 等未登记厂商返回 `STATE_PARSER_VENDOR_UNSUPPORTED`。
- `NetconfBackedDriver.get_current_state()` 和 `verify()` 在触碰设备前先选择 production-ready parser。
- 如果 parser 未验证，真实 NETCONF backend 不会先读设备再失败，避免产生“状态读取已成功”的误判。

价值：

- state parser 和 renderer 一样进入 fail-closed 准入模式。
- 真实设备 parser 未完成前，不会有假 state、假 verify 或 skeleton parser 进入生产路径。
- 后续接 Huawei/H3C parser 时只需要把对应 parser 标记为 `production_ready = True`，并补真实设备验证记录。
