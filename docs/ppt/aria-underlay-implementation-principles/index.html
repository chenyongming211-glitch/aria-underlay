从业务 intent 到 NETCONF 配置下发：事务、状态、厂商适配和离线验收闭环。

这一页先把培训目标说清楚：我们不是讲一个单点功能，而是讲 Aria Underlay 怎样把网络配置变成一个可验证、可恢复、可审计的工程系统。

今天会沿着真实执行链路讲：业务输入怎么进入 Rust Core，为什么 Python Adapter 只负责设备动作，事务日志和 shadow state 为什么是核心，H3C NETCONF 怎么通过 renderer、parser 和 offline acceptance 做闭环。

最后会说明边界：哪些能力已经可以离线证明，哪些必须等真实交换机验收后才能提升为生产就绪。

这页给大家一个路线图。培训的重点不是 API 怎么调，而是每个模块为什么这样设计。

最关键的概念有两个：第一，Rust Core 负责最终事务语义，不能让适配器自己决定成功还是失败；第二，状态必须能被重复计算和恢复，所以 journal、shadow、diff、verify 是一条线。

如果听完以后大家能解释“为什么不能用 FullReplace 隐式删除”、“为什么没有真实交换机时只能 fixture/offline 验证”，这次培训就达标了。

vendor profile
产品核心目标 把网络配置下发从“脚本动作”升级成“有状态、有事务、有恢复证据的控制面”。

这里要强调：我们做的不是一个 NETCONF 命令包装器。真正的问题是，配置系统一旦开始修改设备，就必须能回答三个问题：现在做到哪一步了、失败以后能不能恢复、恢复以后还能不能证明状态一致。

所以产品目标不是“发 XML 成功”，而是发之前能验证、发的时候能锁住、发完能读回、失败能恢复、人工介入能审计。

这也是后来移除 FullReplace 的原因。删除必须显式表达，例如 delete VLAN、delete ACL、delete interface config，而不是靠 desired state 缺失来推断。

architecture.rs

脑手分离：Rust 做控制面，Python 做设备动作

Rust Core

事务语义、规划、diff、journal、shadow、recovery、worker、运维 API。

intent validation planner diff / normalize transaction journal endpoint lock recovery
gRPC / protobuf DesiredDeviceState Prepare / Commit / Verify / Recover

Python Adapter

厂商 renderer/parser、NETCONF session、mock backend、dry-run preflight。

H3C renderer state parser ncclient mock NETCONF offline acceptance

这个架构的核心是“脑手分离”。Rust 是脑子，决定事务怎么走、哪些变更允许、失败时系统最终应该是什么状态。Python 是手，负责连接设备、渲染 XML、执行 NETCONF 操作和读回解析。

为什么不让 Python 决定事务结果？因为 Python 更贴近设备，会遇到各种厂商细节。如果让它自己推断成功失败，Rust journal 就失去权威，恢复路径会混乱。

所以 gRPC 边界很重要：Rust 给出明确 action，Python 执行并返回结构化结果。最终 operation status 必须由 Rust 根据 journal、adapter result、verify 和 shadow write 一起决定。

apply_pipeline()

一次 apply 的主路径

1Intent用户声明 create / update / delete
2Validate字段、厂商能力、显式删除冲突
3Plandomain intent 拆成 device desired
4Diffdesired vs shadow 生成 change set
5Preparerenderer 输出 XML，adapter 写 candidate
6Commitconfirmed commit / candidate commit
7Verifyget-config 读回，parser 标准化比对
8Shadow落盘新状态，再写 Committed
关键约束
没有 journal 的配置变更不能进入 running
没有显式 delete intent 的资源不能因为缺失而被删除
adapter RPC 返回成功不等于最终事务成功
verify 不通过时必须回滚或进入 InDoubt

这一页从执行链路看系统。用户输入的是 intent，不是设备命令。intent 先校验，再被 planner 拆成每台设备的 desired state。

diff 阶段只比较 desired 和 shadow，并且现在只支持 MergeUpsert 加显式删除。也就是说，没提到的资源保留，提到 delete 的资源才删除。

真正进入设备前，Rust 会写 journal 并获取 endpoint lock。设备执行完成后，还要 verify 读回。最后顺序也很关键：shadow 写成功后才能把 journal 标记为 Committed，这样才能缩小崩溃窗口。

phase transition 统一入口

生产路径不直接改 phase，必须走 transition_phase 校验状态转移。

InDoubt 不自动清理

不能证明最终状态时阻断后续事务，由人工 force-resolve 并留下审计记录。

事务状态机的价值是把“代码执行到哪了”变成“系统能恢复的证据”。Accepted、Preparing、Prepared、Committing、Committed 是正常路径；失败时会进入 RollingBack、RolledBack 或 InDoubt。

这里有一个原则：不知道就不能假装知道。如果 adapter 崩溃、commit 后 shadow 写失败、或者恢复时无法确认设备状态，系统应该进入 InDoubt，而不是返回 SuccessWithWarning。

ForceResolved 是 break-glass 操作，不是自动修复。它的意义是让运维可以在检查设备后，带着理由解除阻塞，同时保留审计。

state_consistency

三份状态各管一件事

Intent

用户希望系统达到什么状态，包括显式 create / update / delete。

UnderlayDomainIntent

Shadow

Rust 认为设备已经成功落地的标准化状态，是 diff 的基线。

ObservedDeviceState

Observed

从设备 get-config 读回并由 parser 标准化后的实际状态。

GetCurrentState
desired vs shadow生成 change set
observed vs desiredpost-commit verify
observed vs shadowdrift audit

这里讲状态模型。Intent 是输入，shadow 是我们对已落地状态的本地权威记录，observed 是从设备读回的事实。

Diff 用 desired 和 shadow 计算，因为它要知道这次应该发什么变化；verify 用 observed 和 desired 比对，因为它要证明设备最终符合本次期望；drift audit 用 observed 和 shadow 比对，因为它要发现人工改动或设备漂移。

这三份状态不能混用。尤其不能拿 desired 缺失去推断删除，否则系统会把“没说”理解成“删除”，这就是 FullReplace 的风险。

merge_upsert_only

为什么移除 FullReplace，只保留显式删除

FullReplace 的危险

  • 缺失字段会被解释成删除
  • 声明式输入和真实设备差异容易误伤
  • 需要额外红线、审计和防误删机制

MergeUpsert + explicit delete

  • 没提到的资源保持不变
  • 删除必须写进 delete_* intent
  • diff、审计、回滚语义更清楚
intent shape
vlans:                  upsert VLAN
interfaces:             upsert interface access/trunk/description
delete_vlan_ids:         explicit delete VLAN
delete_acl_ids:          explicit delete ACL
delete_interface_names:  explicit delete interface config

这一页解释最近的设计收口。FullReplace 看起来方便，但本质上是隐式删除：用户只要漏写一个资源，系统就可能认为它应该被删除。

现在的方案是只保留 MergeUpsert。创建和更新写在对应 intent 里，删除写在 delete 字段里，比如 delete VLAN、delete ACL、delete interface config。

这样会让客户端多做一步 diff，但核心系统更安全。灾难恢复和 GitOps 仍然能做，只是删除必须由上层工具显式算出来，而不是让 Rust Core 猜。

关键点 renderer 和 parser 必须互相校验：能写进去，也要能读回来；读不懂时必须结构化失败。

厂商适配最容易被低估。不是把命令拼成 XML 就结束了，真正的闭环有三段：renderer、NETCONF backend、state parser。

Renderer 负责把标准模型翻译成 H3C Comware XML。Backend 负责执行 NETCONF 事务动作。Parser 负责把 running XML 读回来，重新变成标准状态，供 verify 和 drift 使用。

这也是为什么没有真实交换机时不能直接说 production ready。离线样本能证明代码边界和基础 XML 结构，但真实设备的 namespace、返回字段、candidate 行为和错误码都还需要验收。

这页讲没有真实交换机时我们怎么推进。offline H3C acceptance 是一个很重要的替代测试，但它不是最终验收。

它做的是完整逻辑闭环：renderer 生成 XML，mock backend 执行 dry-run、prepare、commit、final-confirm，然后吐出类似 H3C running XML，再由 parser 读回，最后 verify。

它的价值是能在 CI 里稳定覆盖命令面和边界，比如显式删除接口配置这次就已经进入 offline acceptance。它的限制也要讲清楚：mock 不能证明真实设备行为。

ops_runtime

运维闭环：日志优先，API 辅助

运行日志

主日志落到 /var/log/aria/aria-underlay.log，用于现场查看错误和关键事件。

operation summary

记录 apply、recovery、GC、drift、force-resolve 的结构化摘要。

worker daemon

负责 GC、drift audit、summary compaction、hot reload status。

收敛原则 内部系统不做 UI、不做外部告警投递、不做 SSO/OIDC/JWT；静态 token 和本地日志满足当前边界。

这里讲产品运维边界。前面我们讨论过，不要把系统过度开发成完整平台。当前是内部系统，所以重点是运行日志、结构化摘要和必要的管理入口。

主日志应该能直接在 /var/log/aria/aria-underlay.log 里看到错误、事务阶段、恢复结果和 worker 事件。operation summary 适合机器读取和后续排障，但不要求运维必须手动查一堆 CLI。

同样，外部 IM、PagerDuty、OIDC、复杂 RBAC 都不是当前需求。这个边界要守住，避免产品复杂度失控。

production_boundary

哪些已经闭环，哪些必须等真机

已能离线证明

  • Rust 事务状态机、journal、shadow、recovery 测试
  • MergeUpsert + 显式删除语义
  • H3C renderer/parser/mock/offline acceptance
  • dry-run preflight、adapter pool、worker daemon 基础闭环

必须真机验证

  • H3C 真实 NETCONF XML schema 和 namespace
  • candidate / confirmed-commit 行为
  • 写入后 readback 是否和 parser 假设一致
  • 非 H3C 厂商 production_ready 提升
上线判断 真实交换机到位后，先采集 running XML、capabilities、YANG schema，再做写入 / readback / cleanup 验收。

这一页是培训里非常重要的风险边界。现在很多东西已经可以离线证明：事务状态机、显式删除、H3C renderer 和 parser、offline acceptance、worker 和日志闭环。

但是厂商生产就绪不能靠 fixture 自封。真实交换机可能返回不同 namespace，某些字段可能缺失，candidate 支持也可能和能力声明不一致。

所以真实设备到位后的第一件事不是继续加功能，而是采集 evidence：running XML、capabilities、YANG schema、写入和 readback 结果。只有这些闭环通过，才能把相关 profile 标成 production ready。

// takeaways

最后记住 5 句话

01配置系统的核心不是下发，而是证明状态。
02Rust Core 是事务权威，Python Adapter 只执行设备动作。
03删除必须显式表达，不能靠 desired 缺失推断。
04journal + shadow + verify 是恢复和审计的基础。
05offline acceptance 能守住代码质量，但不能替代真实交换机验收。

建议会后阅读：README.md、docs/implementation-plan.md、docs/runbooks/offline-h3c-acceptance.md

最后用五句话收尾。第一，配置系统不能只追求“发出去了”，必须证明最终状态。第二，Rust Core 必须保持事务权威。

第三，删除语义要显式，FullReplace 这种隐式删除模式已经移除。第四，journal、shadow 和 verify 是系统能恢复、能审计的基础。第五，离线验收很有价值，但它不是生产就绪的充分条件。

如果大家后续要新增功能，比如新的 H3C 命令面或者新的厂商支持，就按这条链路补齐：intent、validation、planner、diff、renderer、mock、parser、verify、offline acceptance、最后真机验收。
